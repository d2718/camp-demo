Users

There are four classes of user:

Admin
The Admin is responsible for adding, removing, and updating users of the system; adding, removing, and updating courses, and maintaining the academic year calendar.
Boss
The Boss sees an overview of all students' progress, as well as the ability to send automatically-generated email messages to parents. (After Phase II is complete, the Boss will also have access to all student reports.)
Teacher
The Teacher has a comprehensive view of all students assigned to them (by the Admin), and can add and update their goals, Exam scores, and some other grade-relevant information. (After Phase II is complete, the Teacher will also have access to an additional tranche of report-oriented data and the ability to auto-generate, edit, and publish reports.)
Student
The Student has only the ability to view their individual pace calendar, progress, and score.

It is possible for a person to be more than one type of user; it is undoubtedly convenient for at least one Teacher to also be an Admin, and the Boss may also want Admin access as well. All this requires is that a person have a separate uname for each role.

Each role needs to have a valid email address associated with it. This is how a user will set their password initially (and reset it later, if necessary).

User Names

A user's identity is tied to their user name (or, as this system almost universally refers to it, their uname). They are uniquely identified by this uname, and their privileges and capabilities are determined by it. Unames may only contain alphanumeric characters and are case-sensitive.

A uname must match the regular expression /^[a-zA-Z0-9]+$/.

It is recommended to give staff the shortest and most convenient unames, as they will undoubtedly be logging into the system most often. As the system will probably only ever have around 100 users at any given time, no one should end up with a uname that's particularly confusing, obscure, or inconvenient.

Parent Email

It is important that the Student have a valid parent email associated with them, as it is how the parents receive automated messages from the system. (It also ensures the parent can reset the password at any time and thus have access to their student's progress.)

CSV of Students

Students can be added one at a time with the button; this is fine when a single new student enters the system, say, partway through the academic year, but in the fall when all the known students are being added this can be tedious. Students can therefore be added in bulk by uploading their information in a CSV file. The format of the file is as follows:

  • One student record per line.
  • Lines beginning with an octothorpe (#) are considered "comments" and ignored.
  • Unlike when uploading student goals, columns may not be elided.
  • Column order is:
    user name, last name, rest of name, student email, parent email, teacher uname

Courses

Course Symbols

Each Mathematics course has a unique short chunk of letters and numbers that is used to uniquely identify it. This helps differentiate, for example, Honors Algebra I from the Prentice Hall Algebra 1 textbook from Honors Algebra I from Discovering Algebra.

The symbol, like the uname, is a case-sensitive string of alphanumeric characters. It should be reasonably mnemonic and should make sense as a representation of the course it represents. For example, Algebra I from Prentice Hall Algebra 1 has historically been pha1; the honors version of the same course has been pha1h.

A course symbol must match the regular expression /^[a-zA-Z0-9]+$/.

Course Level

The course level is mainly used to sort courses into a roughly chronological order. It should be a decimal number with the integral portion the grade level, and the decimal portion a vague representation of how "difficult" or "challenging" that course is relative to other courses at that grade level. For example, Core Geometry courses have generally fallen in the [10.0, 10.5) range, while Honors Geometry courses have fallen in the [10.5, 11.0) range.

Chapter Titles

Chapters are automatically named upon creation based on their sequence in the book, but you can rename them if you want.

Chapter Subject

A chapter doesn't have to have a subject, but it's convenient for everyone involved (except you, obviously, who has to do the work of adding this information to the database).

Chapter Weight

A chapter's weight is used by the Teacher's function to size the chunk of time allotted to complete that chapter as a goal. By default, when a chapter is inserted (with or ), it is initially assigned a weight of 1.0. It's fine for an entire course to have each chapter weighted equally. When individually weighting chapters in a course, I generally assign a weight by counting the number of sections in the chapter and adding two.

Upload Course File

Adding a Course with the button followed by the button is fairly straightforward. However, if you want to customize the titles, subjects, or weights of each Chapter, this method can rapidly become tedious. Hence, the option to upload all the information about a course in a single text file.

The file format consists of a TOML header specifying data about the overall course, followed by one or more blank lines, followed by a CSV body for specifyhing data about each chapter.

The TOML header has four key/value pairs (all requried):

  • title (a string value)
  • sym (a string value, the course symbol)
  • book (a string value, the texbook for the course, technically required, but it can be blank)
  • level (a floating-point number, the course level)

The CSV portion has four columns: chapter #, weight, title, subject. All the columns except for the chapter # can be left blank, and a suitable default will be supplied. Blank lines and lines beginning with an octothorpe (#) are ignored.

Let's look at a couple of examples. The Core Calculus course file looks thus: 

title = "Core Calculus"
sym = "c"
book = "Calculus: Graphical, Numerical, Algebraic"
level = 13.1

# Last three columns are optional.
# Weights will default to 1.0, titles will default to "Chapter N", and
# subjects will default to nothing.
#
#chapter,   weight,     title,      subject
1,          8,          Chapter 1,  Prerequisites
2,          6,          Chapter 2,  Limits & Continuity
3,          11,         Chapter 3,  Derivatives
4,          8,          Chapter 4,  Applications of Derivatives
5,          7,          Chapter 5,  The Definite Integral
6,          7,          Chapter 6,  Differential Exqns & Modeling
7,          7,          Chapter 7,  Applications of Integration

As you can see, all of the Chapter data is explicitly defined here: chapter weights, titles, and subjects. The 6th Grade Heath Blue book course file has the absolute minimum amount of data specified (only the chapter numbers): 

title = "6th Grade Math"
sym = "6"
book = "Heath Blue Book"
level = 6.0

The Calendar

The Calendar tab has facilities for setting both end-of-semester dates, and specifying all of the "working days" available over the course of the academic year. Working day distribution is necessary for the Teacher's function to work properly, and end-of-semester dates are necessary for semester grades to be calculated correctly.

End-of-semester dates obviously get set with the input fields at the top of the tab; calendar working days are toggled by clicking and dragging on the calendar. Just make sure you click before navigating away from the page.
